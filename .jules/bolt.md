## 2024-05-13 - HashMap Entry Allocation Overhead
**Learning:** Using `HashMap::entry(key.clone()).or_insert(...)` or `HashMap::entry(key.to_path_buf()).or_insert(...)` unconditionally allocates memory for the key even on cache hits. In hot loops like `make_unique_name` in `symbols.rs` or the parser cache, this can lead to massive unnecessary allocation overhead. Attempting to use `if let Some(val) = map.get_mut(&key)` to return a mutable reference followed by an `else { map.insert(...) }` causes borrow checker errors under current NLL due to the mutable borrow extending over the `else` block.
**Action:** When avoiding unconditional allocations for keys on cache hits, use `if !map.contains_key(&key) { map.insert(key.clone(), val); } let val_ref = map.get_mut(&key).expect("just inserted");` for mutable references. For `Arc<T>` or clonable values, use `if let Some(val) = map.get(&key) { val.clone() } else { let val = new_val(); map.insert(key.clone(), val.clone()); val }`.
