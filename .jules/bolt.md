## 2024-05-24 - Rust NLL Check-Then-Insert Borrow Pitfall
**Learning:** When optimizing away allocations using the "if-let get_mut, else insert" pattern (`if let Some(v) = map.get_mut(key) { v } else { map.insert(...); map.get_mut(...) }`), returning the mutable reference directly from the block can trigger an E0499 "cannot borrow `map` as mutable more than once at a time" error. This is because the Rust Non-Lexical Lifetimes (NLL) analysis requires the first mutable borrow (`get_mut`) to remain alive across the entire `if/else` block since its return value is used afterward, conflicting with the `insert` borrow.
**Action:** When keeping the mutable reference alive causes an E0499 error, falling back to a `contains_key` check (`if !map.contains_key(key) { map.insert(...); } let v = map.get_mut(key).unwrap_or_else(|| unreachable!());`) is an acceptable workaround. While it involves an extra hash map lookup on the cold path, it avoids the unconditional heap allocation (such as from `key.to_owned()` and `HashMap::entry`) on the hot path, and safely passes the borrow checker.
