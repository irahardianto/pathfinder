## 2026-05-06 - Avoid redundant String allocations with Borrowing and the Entry API
**Learning:** Double-allocations occur frequently when working with context lines extracted from `grep-searcher` output via `String::from_utf8_lossy`. `to_owned()` is called both by the caller and inside utility functions like `truncate_line` even when limits aren't hit. In nested HashMap groupings, although `HashMap::entry` seems elegant, it takes `K: Into<K>`, meaning it forces a `.clone()` on every loop iteration when working with owned Strings. Using a standard `contains_key` followed by conditional `.insert()` actually saves allocations because it avoids cloning the key on cache hits.
**Action:** When working with potentially lossy UTF-8 text buffers that need string manipulation (trimming, truncation), maintain borrowed `Cow<str>` or `&str` through the pipeline as far as possible, moving the single `.to_owned()` / allocation step to the very end of processing. Use `HashMap::with_capacity` for grouping to avoid reallocations. For hot-path map lookups with owned string keys, avoid `entry(key.clone())` and prefer a `contains_key` check to skip allocating keys that already exist in the map.
