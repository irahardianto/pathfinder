## 2024-05-11 - HashMap Entry Allocations with Borrow Checker Constraints
**Learning:** `HashMap::entry(key.clone()).or_insert(...)` unconditionally allocates on every lookup, creating severe overhead in hot paths (like AST parsing). Replacing it with conditional checks (`contains_key` followed by `get_mut`) successfully avoids allocations, but requires 2 lookups. Due to strict linting rules (`unwrap_used = "deny"`), direct unwrap of `get_mut` will fail CI. Safely appeasing the lint with `if let Some` might demand restructuring loops (e.g. processing values into a `Vec` before map insertion).
**Action:** When replacing `entry()` to avoid allocations in Rust, check for `unwrap_used = "deny"` in the project configuration. For complex updates requiring borrows within loops, process the data first or use `HashMap::entry` correctly with `Cow` or custom lifetime structures if a single-lookup non-allocating method is unavailable.
